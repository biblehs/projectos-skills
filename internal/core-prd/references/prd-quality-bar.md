# PRD Quality Bar

This reference captures a practical PRD standard synthesized from common product documentation patterns and open templates.

## What Strong PRDs Usually Do

- explain the product purpose and release objective
- define the user or use case clearly
- show the core user flow, not only a feature list
- state what is in scope and out of scope
- capture assumptions, constraints, and dependencies
- keep open questions visible
- connect requirements to later design, engineering, and testing work

## What To Bias Toward In This Skill

For this skill system, a strong `PRD_CORE.md` should be concise but should still answer:

1. Why are we building this now?
2. Who is the primary early user?
3. What problem or job is this solving?
4. What is the first validating user flow?
5. What is in scope now and what is intentionally not?
6. What assumptions or constraints could break downstream execution?

## What Not To Do

- do not write an architecture spec inside the PRD
- do not write only promotional language
- do not write only a feature wishlist
- do not hide uncertainty; track open questions explicitly
