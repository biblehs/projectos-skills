# Sprint Manager

Create or refresh `SPRINT_BOARD.md` as the short-horizon execution layer.

This skill should reduce parallelism, not justify more of it.

## Workflow

1. Read the current project master, planning notes, dependency log, and any active module docs.
2. Identify the single sprint objective.
3. Choose a small set of in-progress items that fit the current milestone.
4. Move tempting but non-essential work to backlog.
5. Record blockers explicitly.
6. Write or update `SPRINT_BOARD.md` using the template and rules below.

## Sprint Rules

- Keep one visible center of gravity for the sprint.
- Every in-progress item must map to one module and one milestone.
- Distinguish between core-product work and support-layer work.
- Record blockers in the sprint or dependency log instead of hiding them in chat.
- Prefer low work-in-progress over optimistic overcommitment.

## What To Read

- Start with [sprint-rules.md](references/sprint-rules.md).
- Use [SPRINT_BOARD.template.md](assets/SPRINT_BOARD.template.md) as the default shape.

When the project has no sprint board yet, create one. When it already exists, update it in place and tighten it if it has grown messy or too broad.
