Use this guide when creating or refreshing `execution/WORKTRACK_BOARD.md` and `execution/RESPONSIBILITY_MAP.md`.

## Work Track Definition

A work track is a coherent lane of solo project work with its own goal, boundary, and integration point. It is not just a bucket of random tasks.

## Required Fields For Each Work Track

- name
- type
- purpose
- responsibility note
- current status
- dependencies
- next sync point
- release relevance

## Responsibility Map Should Clarify

- what the user needs to decide for each track
- which tasks are active versus parked
- which surfaces are shared
- how priorities are resolved when tracks conflict

## Healthy Parallelism

- one clear center of gravity
- support lanes that do not constantly disrupt the core lane
- explicit sync points
- visible dependency conflicts

## Unhealthy Parallelism

- several lanes all claiming top priority
- no named center of gravity
- hidden dependencies
- changes landing in shared surfaces without coordination
